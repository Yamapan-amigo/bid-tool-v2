# 入札情報収集ツール V2 — 設計方針書

## 1. システムの目的

> 大島さん（印刷業）が、「今公募中の、自分が取れそうな入札案件」を毎朝スプレッドシートで確認できること。

| 項目 | 内容 |
|------|------|
| 対象 | 印刷・広報・製本関連の公共入札案件 |
| 規模 | 100万〜200万円（300万以下） |
| 地域 | 東京都 + 関東圏 + 国の機関 |
| 状態 | 締切前の公募中案件のみ |
| 費用 | サーバー費用ゼロ |
| 運用 | 毎朝自動取得 → シートで人が最終確認 |

### やらないこと（将来課題）

- 自動応札
- 仕様書の自動読解
- SaaS化（まず自分で使って実績を作る）

## 2. V1の反省点

### 2.1 Gemini依存過多

**問題**: 自治体サイトのHTML→Geminiで案件抽出。入札方式「不明」、日付欠損が多発。GAS版4箇所・Python版6箇所でGemini呼び出し。

**原因**: 開発スピード優先で「Geminiが全部やる」前提の設計。HTML構造を事前調査しなかった。

**教訓**: まずHTMLパターンを分類しパーサーを作る。AIはフォールバック専用。「AIよりアルゴリズムを先に使う」を厳守。

### 2.2 ヘッダー混在による列ズレ

**問題**: V1(13列)、V2(15列)、Python版(16列)の3バージョンが混在。同一シートに書き込んで列ズレ発生。

**原因**: 金額推定機能を後付けで追加し、既存ヘッダーを修正せず新版を追加。

**教訓**: ヘッダーは `config.py` の1箇所のみで定義。変更時は全書き込み関数を同時修正。

### 2.3 官公需APIのフィルタ不足

**問題**: 全国の道路工事・建設案件を大量取得。「官公需ポータル」という発注元名で実名がわからない。

**原因**: APIパラメータ調査不足。「全件取得→後でフィルタ」方針。

**教訓**: APIレベルで最大限絞り込む（Category、地域）。除外キーワードを先に適用。発注元は組織の実名を表示。

### 2.4 カテゴリ分類・おすすめ度の低精度

**問題**: 案件名(title)だけをGeminiに渡して分類→ほぼ全て「その他」「低」。実用的でなかった。

**原因**: 情報量の少ない入力でAI判定を期待しすぎた。

**教訓**: 印刷関連に絞り込み済みなら分類不要。スコアリングはキーワードマッチング+ルールベースで十分。

### 2.5 MVP不足（コード肥大化）

**問題**: GAS 2,939行 + Python 1,693行。金額推定・落札結果収集などの付加機能を基本機能安定前に実装。

**原因**: 「あったらいいな」機能を早期に積み上げた。

**教訓**: Phase 1は「3ソースから取得→シート表示」だけに集中。付加機能は後のPhaseに回す。

### 2.6 e-Tokyoセッション管理の脆弱性

**問題**: JSPのセッションタイムアウト検出はあるがリカバリなし。途中で止まる。

**原因**: ステートフルなJSPアプリの特性を過小評価。GASの手動Cookie管理が不安定。

**教訓**: `requests.Session()` で自動Cookie管理。タイムアウト時は自動再初期化（最大3回リトライ）。

### 2.7 GASの限界

**問題**: GASはHTML解析能力が弱い（BeautifulSoupが使えない）。正規表現も制限あり。

**原因**: 「サーバー費用ゼロ＝GAS」で完結させようとした。

**教訓**: 取得・解析はPython、UI表示はGAS+Spreadsheet。GitHub Actionsで実行すればサーバー費用ゼロを維持できる。

## 3. 技術方針（Python + GAS ハイブリッド）

### アーキテクチャ

```
GitHub Actions (cron: 毎朝 8:00 JST)
  └─ Python スクリプト
       ├─ 1. 官公需API → XML parse → filter
       ├─ 2. e-Tokyo → session → search → HTML parse
       ├─ 3. 自治体サイト → fetch → BeautifulSoup parse
       ├─ 4. 統合 → 重複排除 → フィルタ → スコアリング
       └─ 5. gspread → Spreadsheet 書き込み

GAS (Spreadsheet側)
  └─ UI専用: メニュー、条件付き書式、フィルタ、ドロップダウン
```

### 役割分担

| レイヤー | 担当 | 詳細 |
|---------|------|------|
| データ取得・解析 | Python | API呼び出し、HTML/XMLパース、フィルタ、スコアリング |
| データ書き込み | Python (gspread) | Google Service Account経由でSpreadsheet書き込み |
| UI表示 | GAS | 条件付き書式、ドロップダウン、フィルタ設定のみ |
| 定期実行 | GitHub Actions | 無料枠 2,000分/月（十分） |

## 4. データソース取得戦略

### 4.1 官公需API

- **エンドポイント**: `https://www.kkj.go.jp/api/`
- **認証**: 不要
- **レスポンス**: XML
- **検索パラメータ**:
  - `Query`: キーワード1つずつ（印刷, 製本, 広報誌, パンフレット, チラシ, ポスター, 冊子, 封筒）
  - `Category`: `1`(物品) と `3`(役務) の両方 ← V1は物品のみだった
  - `CFT_Issue_Date`: 直近30日
  - `Procedure_Type`: 制限しない ← V1は一般競争入札のみだった
- **フィルタ（API取得後）**:
  - 除外キーワード先行: 道路, 河川, 建設, 水道, 電気設備, 下水道, 清掃, 警備, 給食
  - 地域: 東京, 神奈川, 埼玉, 千葉 + PrefectureName空欄（中央省庁）
- **パース**: `xml.etree.ElementTree`

### 4.2 e-Tokyo（東京電子自治体共同運営）

- **URL**: `https://www.e-tokyo.lg.jp/choutatu_ppij/ppij/pub`
- **方式**: セッションベースJSP → multipart/form-data POST
- **改善点**:
  - `requests.Session()` でCookie自動管理
  - セッションタイムアウト検出 → 自動再初期化（最大3回）
  - 文字コード: `Windows-31J` → UTF-8変換
- **検索対象**: 23区(101-123) + 26市(201-229) × 8キーワード
- **パース**: 正規表現ベース（V1の `parseProjectRow` を移植）

### 4.3 自治体サイト

- **Phase 1**: スキップ
  - `tama_bid_info_urls.csv` の32自治体中、大多数が「電子調達」= e-Tokyo経由で取得済み
- **Phase 4以降**: 独自HTML型サイト（5自治体程度）の個別パーサーを作成
  - BeautifulSoupベース
  - サイトごとにパーサークラスを分離

## 5. AI使用基準

### 判断フレームワーク

```
構造化データ（API/XML/定形HTML） → アルゴリズム
非構造化 かつ パターン5種以上    → AI検討
バイナリ（PDF）                   → AI（将来課題）
```

### Phase 1: AI不使用

| 処理 | 手法 |
|------|------|
| 官公需XMLパース | `xml.etree.ElementTree` |
| e-Tokyo HTMLパース | 正規表現 + BeautifulSoup |
| 印刷関連フィルタ | キーワードマッチング（含む/除外） |
| スコアリング | ルールベース（キーワード出現 + 入札方式 + 地域 + 締切） |
| 日付変換 | 正規表現 |
| 重複排除 | 案件名+発注元のキー比較 |

### Phase 2以降でAI使用を検討する箇所

| 処理 | 条件 |
|------|------|
| 独自HTML型サイトのパース | 個別パーサーで対応できない場合のみ |
| PDF仕様書の要約 | 将来課題（Phase 6） |
| 曖昧な案件名の分類 | キーワードマッチングで判定不能なもののみ |

## 6. データモデル

### 案件一覧シート（11列）

| 列 | カラム名 | 型 | ソース | 説明 |
|----|---------|-----|-------|------|
| A | 取得日 | date | Python | YYYY-MM-DD |
| B | 案件名 | text | API/HTML | 案件タイトル |
| C | 発注元 | text | API/HTML | 組織名・自治体名（実名） |
| D | 入札方式 | text | API/HTML | 一般競争入札 / 指名競争入札 / 随意契約 等 |
| E | 公告日 | date | API/HTML | YYYY-MM-DD |
| F | 締切日 | date | API/HTML | YYYY-MM-DD |
| G | 案件詳細URL | url | API/HTML | 案件ページへのリンク |
| H | データソース | text | Python | 官公需 / e-Tokyo / 自治体名 |
| I | スコア | number | Python | 1-5（アルゴリズム算出） |
| J | メモ | text | ユーザー | 手動入力 |
| K | ステータス | enum | ユーザー | 未確認 / 確認済 / 応札予定 / 見送り |

### V1からの変更

- 13列 → 11列に削減
- 廃止: 推定金額、仕様書URL（詳細URLから辿れる）、カテゴリ（印刷で絞り済み）、おすすめ度（文字列→スコアに変更）
- 追加: データソース（どこから取得したか明示）

### スコアリングロジック（案）

```python
base = 3
if コアキーワード(印刷, 製本, 冊子, 広報誌) in 案件名:  +1
if 入札方式 == 一般競争入札:                              +1
if 発注元が東京都内:                                      +0.5
if 締切まで7日以上:                                       +0.5
if 除外ワード(システム, 工事, 清掃) in 案件名:            -2
→ clamp(1, 5)
```

## 7. 開発原則

1. **一次情報を取得する** — 元サイトのAPI/HTMLから直接取得。二次情報サイトは使わない
2. **AIよりアルゴリズムを先に使う** — 正規表現・BeautifulSoup・XMLパーサーで対応。無理な場合のみAI
3. **MVP First** — 各Phaseで動くものを出す。「あったらいいな」は後のPhaseに回す
4. **ヘッダーは1箇所で定義** — `config.py` のみ。GAS側は参照するだけ
5. **テストを先に書く** — 各パーサーにはテストHTMLのfixture + pytestを用意

### コーディングルール

- Python 3.12+、型注釈必須
- `dataclass(frozen=True)` でデータモデル定義
- ログは `logging` モジュール（`print()` 禁止）
- 環境変数は `.env` + `python-dotenv`
- フォーマッター: `ruff format`、リンター: `ruff check`
